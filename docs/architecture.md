# CBRS Commerce Registry Scraper — Architecture & Technical Notes

## Overview

This scraper automates searching and downloading commercial inscriptions from the
**Conservador de Bienes Raíces de Santiago (CBRS)** portal at
`https://nuevo-portal.conservador.cl`. It targets the "Índice del Registro de Comercio"
section, which allows searching by razón social (text) or by foja/número/año (FNA),
and downloading the associated document page images.

---

## Architecture: Hybrid curl_cffi + Playwright

The scraper uses a **hybrid approach**:

- **curl_cffi** — handles all HTTP requests (login, search, image download) with
  Chrome TLS fingerprint impersonation (`impersonate="chrome120"`)
- **Playwright** — runs a minimal Chrome browser instance solely for generating
  reCAPTCHA Enterprise tokens via the real Google SDK

### Why This Architecture

The site has two protection layers:
1. **Imperva WAF** — ties session cookies to TLS fingerprint
2. **reCAPTCHA Enterprise** — score-based tokens from the real Google JS SDK

curl_cffi solves the Imperva layer by impersonating Chrome's TLS/JA3 fingerprint.
The browser is still needed for reCAPTCHA because third-party solvers (2captcha,
CapSolver) produce low-score tokens that the server rejects.

### Advantages Over Pure Playwright

| Aspect | Before (Playwright only) | After (curl_cffi + Playwright) |
|---|---|---|
| Image download | base64 round-trip via FileReader | Direct binary download |
| API calls | `page.evaluate(fetch(...))` | Native HTTP via curl_cffi |
| Browser load | Full page + all JS for every call | Page loaded once, only for tokens |
| Debugging | Opaque browser JS | Visible HTTP request/response |
| Resource usage | High (full browser for everything) | Low (browser idle between tokens) |

## Previous Approaches (Failed)

### Attempt 1: httpx + 2captcha (pure HTTP)

**Approach:** Use httpx for all HTTP requests and 2captcha's `recaptcha()` method
with `enterprise=1` to solve reCAPTCHA Enterprise tokens remotely.

**What happened:**
- 2captcha successfully solved the captcha and returned tokens (valid-looking
  `0cAFcWeA...` format, ~2000 chars).
- The CBRS server **rejected every 2captcha token** with HTTP 400:
  ```json
  {
    "code": "intente-mas-tarde",
    "msg": "Se ha detectado un problema, refresque la página e intente nuevamente."
  }
  ```
- This happened for **both** the login endpoint (`/api/v1/auth/login`) and the
  search endpoint (`/api/v1/comercio/indice/texto`).

**Root cause:** The CBRS backend validates reCAPTCHA Enterprise tokens server-side
by sending them to Google's `recaptchaenterprise.googleapis.com` assessment API.
Google returns a **score** (0.0–1.0) and the **action** string. Tokens generated by
2captcha's human workers produce low scores because they don't run the actual
reCAPTCHA Enterprise JavaScript SDK — they solve the challenge in an isolated
environment. The CBRS server likely has a minimum score threshold that 2captcha
tokens cannot meet.

### Attempt 2: httpx with browser JWT (bypass login)

**Approach:** Extract a valid JWT from the real Chrome browser (via Chrome DevTools
MCP), inject it into httpx, and use 2captcha tokens only for search (bypassing
login entirely).

**What happened:**
- Extracted JWT and Imperva WAF cookies from the browser.
- Even with a valid browser JWT, the 2captcha search token was still rejected
  with the same 400 "intente-mas-tarde" error.
- This confirmed the issue is **the captcha token quality itself**, not the
  login flow.

### Attempt 3: httpx with browser-generated reCAPTCHA token

**Approach:** Generate a real reCAPTCHA token in the browser using
`grecaptcha.enterprise.execute()` via Chrome DevTools, then use it in httpx.

**What happened:**
- The browser-generated token is valid (confirmed by successful searches in the
  browser itself).
- But when used from httpx with **httpx-generated WAF cookies** (different from
  the browser's), the server returned HTTP **403 Forbidden**.
- The Imperva WAF cookies (`visid_incap_*`, `incap_ses_*`, `e840d1eb...`) are
  tied to the browser session. Using a different set of WAF cookies causes
  Imperva to block the request before it reaches the application.

**Key insight:** The server has **two layers** of validation:
1. **Imperva WAF** — validates session cookies match the request origin (403 if mismatched)
2. **Application** — validates reCAPTCHA token with Google's API (400 if low score)

To pass both layers, the WAF cookies and the reCAPTCHA token must come from the
**same browser session**. curl_cffi solves this by impersonating Chrome's TLS
fingerprint, allowing it to reuse browser WAF cookies successfully.

## Imperva/Incapsula WAF Details

The CBRS portal is protected by **Imperva (formerly Incapsula)** CDN/WAF.

### WAF Cookies
| Cookie | Purpose |
|---|---|
| `visid_incap_3235956` | Persistent visitor ID (long-lived) |
| `incap_ses_621_3235956` | Session cookie (per-session) |
| `e840d1eb806b59c4ef38a3048c980757` | Fingerprint/tracking cookie |

These cookies are set on the first visit to the site and validated on subsequent
requests. They are tied to the TLS fingerprint, User-Agent, and other browser
characteristics. httpx requests with self-generated WAF cookies are rejected (403)
because httpx has a different TLS fingerprint than Chrome. curl_cffi solves this
by impersonating Chrome's exact TLS/JA3 fingerprint.

### Cookie Sync Strategy

WAF cookies are generated by the Playwright browser (which loads the full page)
and then synced to the curl_cffi session via `RecaptchaTokenGenerator.get_cookies()`.
This ensures both systems share the same WAF session identity.

### Incapsula Resource Script
The page loads `/_Incapsula_Resource?SWJIYLWA=...&ns=1&cb=...` which is
Imperva's client-side bot detection JavaScript. In Playwright, this script runs
but does not block operation since we use a real Chrome browser.

## reCAPTCHA Enterprise Details

### Configuration
- **Site key:** `6Le-eiksAAAAANU-0ITcjxvGfFoHsz40juvUVI_-`
- **Type:** reCAPTCHA Enterprise (score-based, no visible challenge)
- **SDK URL:** `https://www.google.com/recaptcha/enterprise.js?render=<sitekey>`

### Actions Used
| Action | Used For |
|---|---|
| `login` | Authentication (`/api/v1/auth/login`) |
| `indice_com_texto` | Search and ticket validation |

### Token Delivery
The reCAPTCHA token is sent in **two places** for search requests:
1. HTTP header: `recaptcha-token: <token>`
2. JSON body field: `recaptchaToken: "<token>"`

For login, it's only in the header.

### Why the SDK Must Be Manually Injected

When Playwright opens the CBRS site, the Vue.js application loads but the
reCAPTCHA Enterprise SDK (`enterprise.js`) is **not automatically loaded**.
In the real browser (opened manually), the app dynamically loads it. In Playwright,
it doesn't — likely because:

1. The SDK loading is triggered by a Vue component lifecycle event that depends on
   conditions not met in the Playwright context (e.g., Imperva validation state).
2. The CSP meta tag restricts script evaluation, and while `bypass_csp=True` helps,
   some dynamic loading paths may still be affected.

The solution is to **manually inject** the SDK via `document.createElement('script')`
and wait for `grecaptcha.enterprise.ready()` before generating tokens.

## API Flow

### Authentication

```
1. POST /api/v1/home/start          — Get session hint (body: {"preHint":""})
2. grecaptcha.enterprise.execute()   — Generate reCAPTCHA token (action: "login")
3. POST /api/v1/auth/login           — Login with email/password
   Headers: recaptcha-token: <token>
   Body: {"email":"...","password":"..."}
   Returns: {"token":"<JWT>", ...}
   Sets: cbrs_refresh_token (HttpOnly cookie)
```

### JWT Management
- JWT expires in **30 minutes** (`exp` field in payload).
- Refresh via `POST /api/v1/auth/refresh` (uses `cbrs_refresh_token` cookie).
- The `auth_cbrs_token` cookie (JWT wrapped in quotes) is set for image download
  authentication: `auth_cbrs_token="<jwt>"`.

### Search

Both text and FNA searches use the **same endpoint**:

```
POST /api/v1/comercio/indice/texto
Headers: Authorization: Bearer <jwt>, recaptcha-token: <token>
Body (text):  {"texto":"MBX Global","origen":"texto", ...}
Body (FNA):   {"foja":63244,"numero":27964,"ano":2022,"origen":"fna", ...}
Returns: [{id, foja, num, ano, nombreSociedad, acto, ticket, ...}, ...]
```

### Image Download

```
1. POST /api/v1/comercio/indice/fnaTicket   — Validate ticket, get foja/numero/ano
   Body: {"ticket":"<hash>"}
   Returns: {foja, numero, ano, isFna, ticket}

2. POST /api/v1/comercio/indice/img          — Get image references
   Body: <ticket_info from step 1>
   Returns: {numberOfPages, refs: [{pageNumber, dataRef, dataRefThumb}, ...]}

3. GET /api/v1/comercio/indice/img/<uuid>    — Download image
   Auth: via auth_cbrs_token cookie (NOT Bearer header)
   Returns: image/jpeg binary (~100KB per page)
```

**Note:** Images are now downloaded directly as binary via curl_cffi (no more
base64 encoding/decoding through the browser).

## File Structure

```
cbrs/
├── __init__.py         # Module init
├── __main__.py         # Entry point: calls cli.main()
├── cli.py              # CLI argument parsing and commands
├── config.py           # Configuration (.env loading, constants)
├── recaptcha.py        # RecaptchaTokenGenerator (Playwright, minimal)
├── scraper.py          # CBRSScraper (curl_cffi + RecaptchaTokenGenerator)
└── scraper_http.py     # Test harness for HTTP-only experiments (Phases 1-3)
```

### Key Classes

**`RecaptchaTokenGenerator`** (`recaptcha.py`):
- Launches Playwright Chrome with anti-detection measures (headless UA patching)
- Navigates to CBRS site and injects reCAPTCHA Enterprise SDK
- Handles login via browser `fetch()` (avoids WAF TLS fingerprint mismatch)
- Exposes `generate_token(action) -> str`, `login(email, password) -> dict`, and `get_cookies() -> dict`
- Lazy initialization (browser only starts on first use)

**`CBRSScraper`** (`scraper.py`):
- Creates `RecaptchaTokenGenerator` + `curl_cffi.requests.Session`
- Syncs WAF cookies from browser to curl_cffi on first login
- All API calls go through curl_cffi with Chrome TLS fingerprint
- Same public API as before: `search_by_text`, `search_by_fna`, `download_all_images`

## Experimental Test Harness

`scraper_http.py` provides a progressive series of experiments to test if
the browser can be fully eliminated:

- **Phase 1:** Can curl_cffi pass Imperva WAF on its own (without browser cookies)?
- **Phase 2:** Are reCAPTCHA tokens strictly enforced on all endpoints?
- **Phase 3:** Can CapSolver generate accepted reCAPTCHA Enterprise tokens?

Run with: `python -m cbrs.scraper_http phase1|phase2|phase3|all`

If Phase 1 + Phase 3 both pass, the browser can be completely removed.

## Rate Limits

The CBRS portal enforces a **daily query limit** per account. After exceeding it:
```json
{
  "code": "err-limite",
  "msg": "Se han agotado las consultas disponibles por hoy."
}
```

The exact limit is unknown but appears to be around 15–20 searches per day.
The scraper supports **multi-account rotation** to extend the daily quota.

## Experiment Results

### Phase 1: curl_cffi WAF Bypass — PASS

curl_cffi with `impersonate="chrome120"` successfully passes Imperva WAF:
- Gets all 3 WAF cookies (`visid_incap_*`, `incap_ses_*`, fingerprint)
- `/api/v1/home/start` returns 200 with valid session hint
- No browser cookies needed — curl_cffi's TLS fingerprint is sufficient

### Phase 2: reCAPTCHA Enforcement — Strict on Login

| Endpoint | Captcha | Status | Verdict |
|---|---|---|---|
| `/home/start` | None | 200 | No captcha needed |
| `/auth/refresh` | None | 401 | No captcha, just needs refresh token cookie |
| `/auth/login` | None | 400 | Captcha strictly enforced |
| `/auth/login` | Empty `""` | 400 | Captcha strictly enforced |

Login requires a real reCAPTCHA token. Token refresh only needs the
`cbrs_refresh_token` cookie (no captcha), so after one login we can refresh
indefinitely.

### Phase 3: CapSolver — Not Tested

CapSolver was not tested (requires API key and account funding). If it works,
the browser dependency can be fully eliminated.

## Deployment on Headless Linux

The scraper auto-detects the browser: uses system Google Chrome if available,
falls back to Playwright's bundled Chromium otherwise.

### Option A: System Chrome (recommended — better reCAPTCHA scores)

```bash
# Install Chrome
wget -q -O - https://dl.google.com/linux/linux_signing_key.pub | sudo apt-key add -
echo "deb http://dl.google.com/linux/chrome/deb/ stable main" | \
    sudo tee /etc/apt/sources.list.d/google-chrome.list
sudo apt update && sudo apt install -y google-chrome-stable
```

### Option B: Bundled Chromium (no root needed)

```bash
# Install Playwright's bundled Chromium + system deps
pip install playwright
playwright install chromium
playwright install-deps chromium  # needs sudo on Debian/Ubuntu
```

### Verify

```bash
python -m cbrs -v search -q "test"
```

You should see either `Using system Chrome for reCAPTCHA` or
`System Chrome not found, using bundled Chromium` in the debug output.

### curl_cffi

curl_cffi requires no special system dependencies beyond what pip installs.
It bundles its own libcurl with BoringSSL for TLS fingerprint impersonation.

## Troubleshooting

### "Se ha detectado un problema, refresque la página e intente nuevamente"
This is the `intente-mas-tarde` error. Two possible causes:

1. **Headless Chrome detected (primary):** Chrome's headless mode sets the UA to
   `HeadlessChrome/...`. reCAPTCHA Enterprise detects this and assigns low scores.
   The scraper patches the UA automatically. If this breaks, run with `-v` and check
   for "Patched headless UA" in logs, or test with `--no-headless`.

2. **reCAPTCHA token rejected (secondary):** The `grecaptcha.enterprise.execute()`
   call returned a stale or invalid token. Check that the SDK was properly injected
   and that the sitekey hasn't changed.

Login is done via `fetch()` inside the browser (not curl_cffi) to avoid TLS
fingerprint mismatches with Imperva WAF cookies.

See `docs/intente-mas-tarde-debugging.md` for the full debugging history.

### "Se han agotado las consultas disponibles por hoy"
Daily query limit reached. Add more accounts to `.env` or wait until the next day.

### reCAPTCHA SDK not loading
The SDK must be manually injected. If `grecaptcha.enterprise.execute` is not a
function after injection, wait for `grecaptcha.enterprise.ready()` — the SDK
needs time to initialize after the script loads.

### Playwright CSP errors
Ensure `bypass_csp=True` is set on the browser context. The CSP meta tag in the
HTML blocks `unsafe-eval` which Playwright needs internally.

### HTTP 403 from Imperva
The WAF session cookies are invalid or mismatched. The scraper syncs cookies from
the browser to curl_cffi automatically. If this fails, the browser may not have
loaded properly — check that Playwright's browser is installed.

### Image download returns HTML instead of JPEG
The `auth_cbrs_token` cookie may not be set correctly. It must be the JWT wrapped
in quotes: `auth_cbrs_token="<jwt>"`.

### System Chrome not found on headless server
Install Google Chrome or use Playwright's bundled Chromium. See
"Deployment on Headless Linux" above.
